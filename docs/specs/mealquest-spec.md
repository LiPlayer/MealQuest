# MealQuest 商业化落地规范（V14.0）

> 文档定位：本文件是 MealQuest 的产品与商业规范真源（Source of Truth）。
> 适用范围：`MealQuestServer`、`MealQuestMerchant`、`meal-quest-customer` 三端协同建设。

## 0. 版本与治理

- 版本：V14.0（五阶段主线 + 全局价值优化口径）
- 首发区域：中国大陆
- 目标客群：单店与小连锁餐饮商户
- 商业主轴：支付抽佣为主，订阅与增值服务为辅
- 变更规则：任何功能、策略、风控或合规变更，必须先更新本规范，再进入研发。

### 0.1 当前生效口径（V14.0）

- 当前主线目标：优化长期全局价值（Global Value），而非单次转化。
- 当前主线分类：Customer Lifecycle 五阶段（Acquisition、Activation、Engagement、Expansion、Retention）。
- 价值函数默认权重：`0.5 * CustomerLTV30 + 0.3 * MerchantNetProfit30 + 0.2 * PlatformProfit30 - MarketingCost`。
- 当前策略合同主字段：`stage`、`objective`、`decisionSignals`、`gameSupport`。
- Mini-Game 当前定位：互动能力底座，服务五阶段，不作为独立策略主分类。
- 执行治理不变：预算/风险/毛利红线 + 人工确认 + 审计追溯 + 熔断。
- 本文中“六策略族”相关章节为 Legacy 归档，仅用于历史追溯，不作为当前执行真源。

## 1. 北极星目标与核心原则

### 1.1 北极星目标

构建一个面向中小餐饮的“私域经营操作系统”：
- 对商户：降低运营复杂度，在可控风险下稳定提升复购与毛利。
- 对顾客：在支付与互动中获得“即时价值 + 持续资产感”。
- 对平台：建立可复制、可审计、可规模化的商用交付体系。

### 1.2 不变原则

- 无请求不决策：系统不替商户做未授权决策。
- 无确认不执行：营销与资金相关动作必须人工确认后执行。
- 利润优先于活跃：所有策略先满足毛利保护，再追求增长。
- 先闭环后扩张：先跑通最小商用闭环，再扩展策略种类。

## 2. ICP、场景与价值主张

### 2.1 ICP（Ideal Customer Profile）

- 门店规模：单店、2-20 家小连锁
- 经营特点：高频交易、低 IT 能力、对现金流和毛利敏感
- 决策人：老板或店长，结果导向，不接受高学习成本系统

### 2.2 核心场景（JTBD）

- 商户：
  - 我需要用最低运营成本，把到店流量转成可复购用户。
  - 我需要在“促销有效”与“补贴不过线”之间可控平衡。
- 顾客：
  - 我希望扫码即可完成互动与支付，且每次消费都有可感知收益。

### 2.3 平台价值主张

- 做支付与营销闭环，不做点餐、后厨、进销存。
- 用 AI + 治理机制把“运营意图”转成“可执行动作”，并保留审计链路。

## 3. 商业模式与单元经济

### 3.1 收入结构

- 主收入：支付抽佣（按实付交易额计费）
- 辅收入：
  - 高阶 AI 运营包（高级策略、画像、经营建议）
  - 发票自动托管通道服务费
  - 多店协同增强能力包

### 3.2 成本结构

- 可变成本：支付通道、消息通知、AI 推理、云资源
- 风险成本：营销补贴损失、欺诈套利损失、系统故障赔付

### 3.3 商业约束

- 任何营销动作必须通过“预算红线 + 风险红线 + 毛利红线”校验。
- 无法通过校验的策略可以建议，不可执行。

## 4. 产品范围（In/Out）

### 4.1 In Scope（当前版本必须实现）

- 商户端：登录、开店、经营驾驶基础视图、老板端 Agent（查账/发票问答 + 策略提案确认）、紧急停机。
- 顾客端（即小程序端）：扫码入店、Welcome 触达、游戏营销触达/互动、资产展示、支付核销、账本与发票查询。
- 服务端：认证、支付、发票、隐私、策略治理、执行审计、多租户隔离。
- 策略范围（当前生效）：五阶段生命周期闭环（Acquisition、Activation、Engagement、Expansion、Retention）+ Mini-Game 互动能力底座。

- Merchant entry QR: must support generating fixed store QR and distributing via preview/save/share.

### 4.2 Out of Scope（当前版本明确不做）

- 点餐流程、后厨 KDS、桌台管理
- 采购、库存进销存、供应链系统
- 顾客侧非小程序技术路线作为首版交付形态
- 顾客侧 Agent 与店长侧 Agent 的商用开放

## 5. 产品核心形态与资产系统

### 5.1 五类资产层

- 现金账户层（聚宝金库）：本金与赠送金分账，支持退款回溯扣减
- 权益资产层（口福红包）：可核销价值凭证（单品券/无门槛券/折扣权益）
- 叙事资产层（碎银）：互动驱动与品牌记忆载体，强化留存
- 叙事资产层（碎片）：用于成长与合成路径，强化留存
- 游戏收集资产层（游戏收集物）：用于解锁、触达、资格判定与奖励路由，不直接抵扣支付

### 5.2 顾客端产品形态

- 入口形态：扫码入店后快速进入“可支付、可互动、可见资产”首页
- 首页形态：围绕“当前可用资产 + 当前可参与互动 + 当前可执行支付”
- 账户形态：账本、发票、资产变化可自助查询
- 体验目标：顾客每次到店都能明确感知“支付价值 + 资产沉淀”

### 5.3 老板端产品形态

- 入口形态：手机号登录，首登即完成开店并进入经营页
- 经营形态：看板看结果、Agent 给建议、审批控风险、回放看过程、停机控风险
- 操作形态：关键动作可确认、可撤回、可追溯
- 体验目标：老板在 10 分钟内看懂经营状态并完成一次策略执行
- 首版追溯基线：看板需提供最近 24 小时 payment->ledger->invoice->audit 一致性摘要，作为执行回放前置视图

- Entry QR operation: owner can open a dedicated page to generate a fixed store entry QR and distribute it offline.

### 5.4 五类资产-互动绑定矩阵

| 资产层 | 互动来源（Source） | 消耗/流转（Sink） | 商业约束 |
| --- | --- | --- | --- |
| 聚宝金库（现金账户） | 充值、支付退款回补、赠送金入账 | 支付核销、本金/赠送金回溯扣减 | 本金赠送金分账、退款回溯、不可越权扣减 |
| 寻味碎银（叙事资产） | 候餐小游戏结算、首绑互动奖励 | 支付零头抵扣、碎片兑换 | 单局上限、日上限、账号频控 |
| 食福碎片（叙事资产） | 小游戏里程碑、碎银兑换、Welcome 触达奖励 | 合成口福红包、过期分解为碎银 | 配方校验、频控、幂等结算 |
| 口福红包（权益资产） | 碎片合成、Welcome 策略发放 | 支付核销、订单减免 | 预算/库存校验、防重复核销 |
| 游戏收集资产（Collectible） | 新游发布触达、到店互动、支付后掉落策略 | 解锁游戏内容、触发后续营销资格、站内展示沉淀 | 不直接抵扣支付、频控、去重、防通胀 |

### 5.5 体验红线

- 支付主链路优先：互动能力异常不得阻断支付与核销
- 资产可追溯：每次发放、合成、核销必须可被追溯
- 价值可感知：顾客必须看得见资产变化，老板必须看得见经营结果

### 5.6 跨端布局与导航骨架（冻结约束）

- 商户端 IA 骨架固定为：看板、Agent、审批中心、执行回放、风险控制；后续 Step 可填充功能，不可频繁改路由骨架。
- 顾客端 IA 骨架固定为：startup 扫码入店、index 资产首页、account 账票中心；新增能力优先在既有骨架内扩展。
- 布局风格约束：首版采用现代化轻量科技风，强调“信息层级清晰 + 高可读 + 可触达主链路”。
- 导航稳定性约束：任一从商户主入口打开的二维码页必须可安全返回主入口，不得因栈状态异常造成崩溃。
- 布局前置不代表功能前置：占位页仅用于固定信息架构，不视为对应 Step 功能已交付。

## 6. 首版商用闭环定义（MVP，五阶段主线 + Mini-Game 底座）

### 6.1 闭环路径

扫码入店 -> 首次身份绑定 -> Welcome 触发 -> 游戏营销触达/互动 -> 到店支付核销 -> 支付后策略掉落 -> 资产沉淀 -> 复购触达

- Merchant generates and posts store entry QR -> customer scans to enter -> identity binding -> interaction -> payment and asset accrual.
- QR payload boundary (MVP): plain `merchantId` only, validated by server-side merchant existence check; signed payload and one-time short code are out of scope in this release.
- 五阶段按场景编排，不要求单次顾客链路触发全部阶段策略。

### 6.2 拉新礼包（Welcome，Acquisition 子场景）规范

- 目标：以可控补贴换取可验证首单与短周期复购
- 归属：五阶段中的 Acquisition 阶段
- 触发：门店级“首绑成功”事件
- 条件：通过同人识别、预算、库存、频控、风险评分校验
- 执行：默认建议 + 人工确认；紧急场景可人工覆盖并自动过期
- 边界：单客补贴上限、日预算上限、活动总预算上限
- 要求：每次执行都可解释、可审计、可回放
- 可见性：顾客端需反馈命中/未命中结果；商户端需展示 24h 命中/拦截与拦截原因摘要；两端口径一致

### 6.3 游戏营销互动与触达（Game Marketing）规范

- 目标：提升到店互动深度、支付后资产沉淀与短周期复购
- 归属：Mini-Game 互动能力底座（非独立策略阶段）
- 定位：与五阶段策略并行协同，通过平台游戏库和门店策略编排驱动增长
- 形态：单人轻量回合制（无排行、无社交对战）
- 奖励：可发放碎银、碎片、游戏收集资产；红包来源于策略发放或碎片合成
- 营销编排：游戏库由平台统一维护，门店通过策略投放与解锁规则控制顾客可见与可得
- 支付后路由：支付成功后奖励可在碎银/游戏收集资产间策略化路由，并保持账务审计一致
- 降级：游戏营销能力异常时降级为静态互动与默认奖励，不阻断支付链路

### 6.4 顾客旅程（执行前必须对齐）

- 新客：扫码 -> 登录/绑定 -> 看到 Welcome 与游戏触达 -> 完成支付 -> 资产更新可见
- 老客：入店即看到可用资产与活动 -> 快速支付 -> 查询账本/发票
- 异常：网络波动、互动失败时仍可继续支付；后续资产状态可补偿更新

### 6.5 老板旅程（执行前必须对齐）

- 首登：登录 -> 开店 -> 看到初始经营状态
- 日常：查看看板 -> 与 Agent 对话明确目标 -> 确认策略 -> 观察结果 -> 必要时审批或停机
- 异常：发现风险信号时可一键停机，并通过回放定位问题

## 7. AI Agent 与策略耦合治理（产品层）

### 7.1 AI Agent 核心定位

- AI Agent 是老板端的核心经营入口，不是可有可无的附加功能。
- AI Agent 的职责是把“经营意图”转为“可确认、可执行、可复盘”的行动路径。
- AI Agent 不是经营决策主体，最终决策权始终属于老板。

### 7.2 老板端 Agent 能力范围（首版）

- 查询协作：支持老板在会话中查询账务与发票经营事实。
- 策略协作：支持老板在会话中生成策略提案卡，并执行同意/驳回。
- 覆盖范围：首版仅开放老板端 Agent；顾客侧与店长侧 Agent 不在当前商用范围。

### 7.3 会话式经营形态

- 交互形态：老板以自然语言表达目标、约束、问题。
- 模式形态：同一会话可承载“经营问答”与“策略提案”两类协作。
- 协同形态：看板负责“发生了什么”，Agent 负责“下一步做什么”。

### 7.4 查询协作治理

- 查询结果必须基于可验证经营数据，不输出黑盒估算结论。
- 查询必须遵循门店授权边界，不允许越权跨店读取。
- 查询能力异常不得阻断支付、核销与账务主链路。

### 7.5 策略提案协作治理

- Agent 命中策略意图后生成提案卡，老板在 UI 完成同意或驳回。
- 提案卡以业务可读信息为主，不向老板暴露复杂算法参数。
- 提案协作需保持可追溯：提案输入、提案结果、老板决策均可回放。
- 首版必须提供老板手动配置策略参数的能力，且可在同一入口查看 Agent 推荐参数并选择应用。
- Agent 可基于全局效益最优生成参数建议，但不得绕过老板确认直接发布执行。

### 7.6 与策略引擎的耦合原则

- 首版策略商用以五阶段闭环为主；Agent 设计需统一承接五阶段策略意图。
- 候选策略建议采用全局最优思想进行排序和推荐。
- 预算、风险、毛利红线是执行硬门；超线策略可建议但不可执行。
- 五阶段统一采用“可配置模型”：`triggerEvent`、`eligibility`、`action`、`governanceGates`、`objective`、`decisionSignals` 可被商户配置或由 Agent 给出建议后应用。
- 任一配置变更均需进入统一审批与审计链路，保持“无确认不执行”。

### 7.7 主动协作与提醒

- Agent 可对关键经营事件进行主动提醒，保持“少打扰、强价值”原则。
- 提醒应以资金安全、风控异常、关键经营窗口为优先。
- 非营业时段默认抑制非紧急提醒。

### 7.8 失效降级与边界

- Agent 异常时，系统必须降级为基础看板与人工操作流程。
- Agent 异常不得阻断支付、核销、发票、账务等主链路。
- 必须具备人工一键熔断能力，防止策略失控。

## 8. 商业 KPI 与发布门槛（硬指标）

### 8.1 核心 KPI

- `CustomerLTV30`
- `MerchantNetProfit30`
- `PlatformProfit30`
- `Retention30`
- `GlobalValueIndex`
- `SubsidyWasteProxy`
- 支付成功率 >= 99.5%
- 风险损失（套利/欺诈）占 GMV <= 0.3%

说明：
- 上述为当前发布判定主口径。
- 原 7 日转化/核销/参与率等指标保留为运营观测指标，不再作为主发布门。

### 8.2 上线 Go/No-Go 门

- 业务门：`GlobalValueIndex` 达标或具备可验证改善趋势
- 技术门：核心链路稳定，关键路径无阻断
- 风控门：预算、库存、风控约束全部生效
- 合规门：隐私、日志留存、发票流程符合首发区域要求

## 9. 安全、隐私与合规

### 9.1 资金安全

- 本金与赠送金严格分账
- 退款执行回溯扣减，避免赠送权益套利

### 9.2 隐私合规

- 最小化采集，仅采业务必要信息
- 交易数据与行为数据分级留存
- 支持账户注销并执行非交易数据删除

### 9.3 合规运营

- 发票支持手动与托管双轨
- 高风险动作必须审计可追溯

---

## 10. 风险清单与应对

- 套利风险：同人识别 + 风险评分 + 频控 + 审计追责
- 小游戏刷分风险：设备指纹 + 局次频控 + 异常分数熔断
- 资产通胀风险：碎银/碎片发放上限 + 超限降级
- 游戏收集物通胀风险：收集物发放频控 + 去重幂等 + 上限保护
- 毛利侵蚀：预算熔断 + 单客上限 + 活动止损
- 策略失控：熔断开关 + 自动过期 + 审批机制
- 可用性风险：限流、重试、降级、值守机制

## 11. Legacy 归档：六策略族框架与先行验证（历史追溯）

> 本章为历史归档内容：用于追溯 V13.x 阶段设计与验收证据，不作为 V14 当前执行真源。

### 11.1 六策略族框架

- 拉新获客（Acquisition）：以首单转化与新客有效率为核心，首版子域为 Welcome。
- 促活提频（Activation）：以到店频次恢复与互动活跃为核心。
- 提客单/去库存（Revenue）：以客单提升与库存效率为核心。
- 召回留存（Retention）：以沉默人群回流与周期复购为核心。
- 社交裂变（Social Viral）：以有效邀请与低作弊扩散为核心。
- 小游戏运营（Mini-Game Ops）：以小游戏触达、互动结算、资产掉落和后续复购资格为核心。

### 11.2 策略合同与治理统一要求

- 六策略族统一策略合同字段（Legacy）：`strategyId`、`family`、`triggerEvent`、`eligibility`、`action`、`governanceGates`、`customerVisibility`、`merchantVisibility`、`auditTrace`。
- 六策略族复用同一治理红线：预算、风险、毛利、审批、可追溯、熔断。
- 不同策略族可使用不同评估窗口，但必须保持可解释、可回放、可审计。
- 六策略族必须运行在同一通用策略引擎内核上，禁止按策略族复制分叉执行链路。
- 六策略族参数必须支持“老板手动配置 + Agent 建议辅助”，但 Agent 建议不等同于自动执行。
- 每族样例需冻结默认口径（默认触发、默认约束、默认可见性），同时保留在策略合同内可配置扩展能力。

### 11.3 每族一策先行验证清单（首版必须）

- `ACQ_WELCOME_FIRST_BIND_V1`（Acquisition）：新客首绑欢迎权益（实现映射：`acquisition_welcome_gift` 模板默认分支）。
- `ACT_CHECKIN_STREAK_RECOVERY_V1`（Activation）：低活跃顾客回店连签激活（实现映射：`activation_checkin_streak_recovery` 模板默认分支）。
- `REV_ADDON_UPSELL_SLOW_ITEM_V1`（Revenue）：慢销品加购激励与客单提升。
- `RET_DORMANT_WINBACK_14D_V1`（Retention）：14 天沉默顾客召回。
- `SOC_REFER_DUAL_REWARD_FIRST_PAY_V1`（Social Viral）：有效邀请首单双向奖励。
- `GMO_NEW_GAME_UNLOCK_DROP_V1`（Mini-Game Ops）：新游解锁与收集物掉落。

Activation 样例默认口径（首版冻结）：
- 触发：复用 `USER_ENTER_SHOP` 事件执行入口，不新增独立策略事件链路。
- 判定窗口：低活跃阈值 `inactiveDays >= 7`，连签阈值 `checkinStreakDays >= 3`（按自然日去重并连续计数）。
- 激励与频控：奖励参数由老板确认后生效，默认频控窗口为 7 天最多命中 1 次。
- 治理硬门：任一策略参数不完整（含奖励参数缺失）时，不得进入发布与执行流程。
- Agent 边界：Agent 可基于全局价值最优给出参数提案，但必须经老板确认后才可执行。

Revenue 样例默认口径（首版冻结）：
- 触发：默认 `PAYMENT_VERIFY` 执行入口，策略合同内允许配置 `triggerEvent` 以支持后续扩展。
- 库存口径：慢销库存采用策略库存池（`inventory_lock_v1`）管理，不依赖 ERP/进销存系统。
- 配置方式：老板可手动配置 Revenue 参数；Agent 可给出全局最优建议参数供老板应用。
- 治理硬门：任一参数不完整或红线超限时不得进入发布与执行流程。
- 可见性：商户端与顾客端均需看到命中/拦截反馈，并与账务审计口径一致。

Retention 样例默认口径（首版冻结）：
- 触发：复用 `USER_ENTER_SHOP` 事件执行入口，不新增独立策略事件链路。
- 判定窗口：沉默阈值 `inactiveDays >= 14`，且仅对老客生效（`isNewUser = false`）。
- 激励与频控：默认平衡券包；默认频控窗口为 14 天最多命中 1 次。
- 治理硬门：任一策略参数不完整（含沉默阈值缺失）时，不得进入发布与执行流程。
- 可见性：商户端与顾客端均需看到命中/拦截反馈与原因，且口径一致。

### 11.4 先行验证硬约束

- 每族样例都必须形成三端可执行闭环：触发 -> 判定 -> 执行 -> 可见 -> 审计。
- 每族样例都必须证明不破坏支付、核销、账务、发票主链路稳定性。
- 任何超越治理红线的动作不允许执行，只允许输出建议说明。
- 样例进入默认必过门必须遵循 roadmap 指针顺序；未进入当前 `doing` Step 的样例仅可在独立 step suite 中验证。

## 12. 执行文档边界

- 本规范用于定义“做什么、做到什么程度、为什么做”。
- 本规范不承载接口字段、错误码、测试命令、排障入口等实现细节。

- 术语约定：本规范中的“顾客端”在首版执行中等价于“小程序端”。
- 主线/归档约定：V14 当前执行以本规范主章节为准，Legacy 章节仅作历史追溯引用。
