# MealQuest 商业化落地规范（V13.4）

> 文档定位：本文件是 MealQuest 的产品与商业规范真源（Source of Truth）。
> 适用范围：`MealQuestServer`、`MealQuestMerchant`、`meal-quest-customer` 三端协同建设。

## 0. 版本与治理

- 版本：V13.4（补充 Agent-策略耦合业务定义）
- 首发区域：中国大陆
- 目标客群：单店与小连锁餐饮商户
- 商业主轴：支付抽佣为主，订阅与增值服务为辅
- 变更规则：任何功能、策略、风控或合规变更，必须先更新本规范，再进入研发。

---

## 1. 北极星目标与核心原则

### 1.1 北极星目标

构建一个面向中小餐饮的“私域经营操作系统”：
- 对商户：降低运营复杂度，在可控风险下稳定提升复购与毛利。
- 对顾客：在支付与互动中获得“即时价值 + 持续资产感”。
- 对平台：建立可复制、可审计、可规模化的商用交付体系。

### 1.2 不变原则

- 无请求不决策：系统不替商户做未授权决策。
- 无确认不执行：营销与资金相关动作必须人工确认后执行。
- 利润优先于活跃：所有策略先满足毛利保护，再追求增长。
- 先闭环后扩张：先跑通最小商用闭环，再扩展策略种类。

## 2. ICP、场景与价值主张

### 2.1 ICP（Ideal Customer Profile）

- 门店规模：单店、2-20 家小连锁
- 经营特点：高频交易、低 IT 能力、对现金流和毛利敏感
- 决策人：老板或店长，结果导向，不接受高学习成本系统

### 2.2 核心场景（JTBD）

- 商户：
  - 我需要用最低运营成本，把到店流量转成可复购用户。
  - 我需要在“促销有效”与“补贴不过线”之间可控平衡。
- 顾客：
  - 我希望扫码即可完成互动与支付，且每次消费都有可感知收益。

### 2.3 平台价值主张

- 做支付与营销闭环，不做点餐、后厨、进销存。
- 用 AI + 治理机制把“运营意图”转成“可执行动作”，并保留审计链路。

## 3. 商业模式与单元经济

### 3.1 收入结构

- 主收入：支付抽佣（按实付交易额计费）
- 辅收入：
  - 高阶 AI 运营包（高级策略、画像、经营建议）
  - 发票自动托管通道服务费
  - 多店协同增强能力包

### 3.2 成本结构

- 可变成本：支付通道、消息通知、AI 推理、云资源
- 风险成本：营销补贴损失、欺诈套利损失、系统故障赔付

### 3.3 商业约束

- 任何营销动作必须通过“预算红线 + 风险红线 + 毛利红线”校验。
- 无法通过校验的策略可以建议，不可执行。

## 4. 产品范围（In/Out）

### 4.1 In Scope（当前版本必须实现）

- 商户端：登录、开店、经营驾驶基础视图、老板端 Agent（查账/发票问答 + 策略提案确认）、紧急停机。
- 顾客端：扫码入店、候餐互动小游戏（Welcome 范围）、资产展示、支付核销、账本与发票查询。
- 服务端：认证、支付、发票、隐私、策略治理、执行审计、多租户隔离。
- 策略范围：首版仅保障“拉新礼包（Welcome）”闭环可商用。

### 4.2 Out of Scope（当前版本明确不做）

- 点餐流程、后厨 KDS、桌台管理
- 采购、库存进销存、供应链系统
- 非 Welcome 策略的全量商用实现
- 顾客侧 Agent 与店长侧 Agent 的商用开放

## 5. 产品核心形态与资产系统

### 5.1 三类资产层

- 现金账户层（聚宝金库）：本金与赠送金分账，支持退款回溯扣减
- 权益资产层（口福红包）：可核销价值凭证（单品券/无门槛券/折扣权益）
- 叙事资产层（碎银/碎片）：互动驱动与品牌记忆载体，强化留存

### 5.2 顾客端产品形态

- 入口形态：扫码入店后快速进入“可支付、可互动、可见资产”首页
- 首页形态：围绕“当前可用资产 + 当前可参与互动 + 当前可执行支付”
- 账户形态：账本、发票、资产变化可自助查询
- 体验目标：顾客每次到店都能明确感知“支付价值 + 资产沉淀”

### 5.3 老板端产品形态

- 入口形态：手机号登录，首登即完成开店并进入经营页
- 经营形态：看板看结果、Agent 给建议、审批控风险、回放看过程、停机控风险
- 操作形态：关键动作可确认、可撤回、可追溯
- 体验目标：老板在 10 分钟内看懂经营状态并完成一次策略执行

### 5.4 三类资产-互动绑定矩阵

| 资产层 | 互动来源（Source） | 消耗/流转（Sink） | 商业约束 |
| --- | --- | --- | --- |
| 寻味碎银（叙事资产） | 候餐小游戏结算、首绑互动奖励 | 支付零头抵扣、碎片兑换 | 单局上限、日上限、账号频控 |
| 食福碎片（叙事资产） | 小游戏里程碑、碎银兑换、Welcome 触达奖励 | 合成口福红包、过期分解为碎银 | 配方校验、频控、幂等结算 |
| 口福红包（权益资产） | 碎片合成、Welcome 策略发放 | 支付核销、订单减免 | 预算/库存校验、防重复核销 |

### 5.5 体验红线

- 支付主链路优先：互动能力异常不得阻断支付与核销
- 资产可追溯：每次发放、合成、核销必须可被追溯
- 价值可感知：顾客必须看得见资产变化，老板必须看得见经营结果

## 6. 首版商用闭环定义（MVP）

### 6.1 闭环路径

扫码入店 -> 首次身份绑定 -> 候餐小游戏互动 -> 触发拉新礼包 -> 到店支付核销 -> 资产沉淀 -> 复购触达

### 6.2 拉新礼包（Welcome）规范

- 目标：以可控补贴换取可验证首单与短周期复购
- 触发：门店级“首绑成功”事件
- 条件：通过同人识别、预算、库存、频控、风险评分校验
- 执行：默认建议 + 人工确认；紧急场景可人工覆盖并自动过期
- 边界：单客补贴上限、日预算上限、活动总预算上限
- 要求：每次执行都可解释、可审计、可回放

### 6.3 候餐小游戏（Welcome 范围）规范

- 目标：提升首绑后互动参与率与短周期复购
- 定位：Welcome 的互动增效器，不作为独立营销策略
- 形态：单人轻量回合制（无排行、无社交对战）
- 奖励：仅发放碎银与碎片；红包来源于策略发放或碎片合成
- 降级：小游戏异常可降级为静态互动，不阻断支付链路

### 6.4 顾客旅程（执行前必须对齐）

- 新客：扫码 -> 登录/绑定 -> 看到欢迎互动与可得奖励 -> 完成支付 -> 资产更新可见
- 老客：入店即看到可用资产与活动 -> 快速支付 -> 查询账本/发票
- 异常：网络波动、互动失败时仍可继续支付；后续资产状态可补偿更新

### 6.5 老板旅程（执行前必须对齐）

- 首登：登录 -> 开店 -> 看到初始经营状态
- 日常：查看看板 -> 与 Agent 对话明确目标 -> 确认策略 -> 观察结果 -> 必要时审批或停机
- 异常：发现风险信号时可一键停机，并通过回放定位问题

## 7. AI Agent 与策略耦合治理（产品层）

### 7.1 AI Agent 核心定位

- AI Agent 是老板端的核心经营入口，不是可有可无的附加功能。
- AI Agent 的职责是把“经营意图”转为“可确认、可执行、可复盘”的行动路径。
- AI Agent 不是经营决策主体，最终决策权始终属于老板。

### 7.2 老板端 Agent 能力范围（首版）

- 查询协作：支持老板在会话中查询账务与发票经营事实。
- 策略协作：支持老板在会话中生成策略提案卡，并执行同意/驳回。
- 覆盖范围：首版仅开放老板端 Agent；顾客侧与店长侧 Agent 不在当前商用范围。

### 7.3 会话式经营形态

- 交互形态：老板以自然语言表达目标、约束、问题。
- 模式形态：同一会话可承载“经营问答”与“策略提案”两类协作。
- 协同形态：看板负责“发生了什么”，Agent 负责“下一步做什么”。

### 7.4 查询协作治理

- 查询结果必须基于可验证经营数据，不输出黑盒估算结论。
- 查询必须遵循门店授权边界，不允许越权跨店读取。
- 查询能力异常不得阻断支付、核销与账务主链路。

### 7.5 策略提案协作治理

- Agent 命中策略意图后生成提案卡，老板在 UI 完成同意或驳回。
- 提案卡以业务可读信息为主，不向老板暴露复杂算法参数。
- 提案协作需保持可追溯：提案输入、提案结果、老板决策均可回放。

### 7.6 与策略引擎的耦合原则

- 首版策略商用以 Welcome 闭环为主；Agent 设计需可扩展至五类策略族。
- 候选策略建议采用全局最优思想进行排序和推荐。
- 预算、风险、毛利红线是执行硬门；超线策略可建议但不可执行。

### 7.7 主动协作与提醒

- Agent 可对关键经营事件进行主动提醒，保持“少打扰、强价值”原则。
- 提醒应以资金安全、风控异常、关键经营窗口为优先。
- 非营业时段默认抑制非紧急提醒。

### 7.8 失效降级与边界

- Agent 异常时，系统必须降级为基础看板与人工操作流程。
- Agent 异常不得阻断支付、核销、发票、账务等主链路。
- 必须具备人工一键熔断能力，防止策略失控。

## 8. 商业 KPI 与发布门槛（硬指标）

### 8.1 核心 KPI

- 新客首绑到首单转化率（7 日） >= 12%
- 拉新补贴成本（每有效新客） <= 12 元
- 拉新礼包 7 日核销率 >= 20%
- 小游戏参与率（首绑后 24 小时） >= 35%
- 碎片合成到红包转化率（7 日） >= 15%
- 营销补贴占 GMV 比例 <= 5%
- 支付成功率 >= 99.5%
- 风险损失（套利/欺诈）占 GMV <= 0.3%

### 8.2 上线 Go/No-Go 门

- 业务门：核心 KPI 达标或具备可验证改善趋势
- 技术门：核心链路稳定，关键路径无阻断
- 风控门：预算、库存、风控约束全部生效
- 合规门：隐私、日志留存、发票流程符合首发区域要求

## 9. 安全、隐私与合规

### 9.1 资金安全

- 本金与赠送金严格分账
- 退款执行回溯扣减，避免赠送权益套利

### 9.2 隐私合规

- 最小化采集，仅采业务必要信息
- 交易数据与行为数据分级留存
- 支持账户注销并执行非交易数据删除

### 9.3 合规运营

- 发票支持手动与托管双轨
- 高风险动作必须审计可追溯

---

## 10. 风险清单与应对

- 套利风险：同人识别 + 风险评分 + 频控 + 审计追责
- 小游戏刷分风险：设备指纹 + 局次频控 + 异常分数熔断
- 资产通胀风险：碎银/碎片发放上限 + 超限降级
- 毛利侵蚀：预算熔断 + 单客上限 + 活动止损
- 策略失控：熔断开关 + 自动过期 + 审批机制
- 可用性风险：限流、重试、降级、值守机制

## 11. 策略扩展方向（产品层）

> 首版以 Welcome 闭环商用为目标。以下为 Agent-策略耦合的扩展框架，用于后续分步落地，不代表当前版本必须一次性实现。

### 11.1 五类策略族框架

- 拉新获客（Acquisition）：以首单转化与新客有效率为核心。
- 促活提频（Activation）：以到店频次恢复与互动活跃为核心。
- 提客单/去库存（Revenue）：以客单提升与库存效率为核心。
- 召回留存（Retention）：以沉默人群回流与周期复购为核心。
- 社交裂变（Social Viral）：以有效邀请与低作弊扩散为核心。

### 11.2 Agent-策略耦合原则

- Agent 统一承接五类策略意图输入，输出业务可读提案。
- 不同策略族复用同一治理红线：预算、风险、审批、可追溯、熔断。
- 不同策略族可使用不同评估窗口，但必须保持可解释与可复盘。

### 11.3 扩展硬约束

- 每新增策略都必须证明对毛利与复购有正向增益。
- 每新增策略都不得破坏支付、核销、账务、发票主链路稳定性。
- 任何超越治理红线的动作不允许执行，只允许给出建议说明。
