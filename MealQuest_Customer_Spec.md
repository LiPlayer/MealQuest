# 餐餐有戏 顾客端产品全景规范文档

> **项目名称**：餐餐有戏 (MealQuest)
> **核心定义**：基于“资产包”逻辑的单页式餐饮应用。
> **文档目标**：本文档描述了产品的**页面层级结构 (Page Hierarchy)** 及**页面详细布局 (Page Layouts)**，明确所有功能的入口路径。

## 第零章：全局技术规范 (Global Technical Specs)

*   **开发框架**: Taro (React) + Tailwind CSS.
*   **分辨率**: 750 x 1624 (iPhone X @ 2x).
*   **小程序适配**: **双端兼容**。本项目仅针对 **微信小程序 (WeChat Mini Program)** 与 **支付宝小程序 (Alipay Mini Program)** 进行开发与测试。不再兼容 H5 端，所有仅用于 H5 的兼容逻辑应被移除。

### 0.1 关键开发约束 (Critical Constraints)

1.  **单位主权 (Unit Sovereignty)**:
    *   **rpx 标准**: 本项目严格遵循小程序 **rpx 布局标准**。不论物理分辨率如何，视口宽度始终固定为 **750 个单位**。
    *   **等比缩放**: 默认 `px` 自动转 `rpx` (由 `designWidth: 750` 驱动)。适用于 90% 的布局、间距、字体。
    *   **物理像素**: 必须使用大写 `PX` 单位（如 `border: 1PX solid #000`），确保 1px 线条在 2x/3x 屏幕下依然保持极致纤细，不被缩放。
2.  **根字号与 Tailwind 映射**:
    *   **锁定**: `app.scss` 必须锁定 `html { font-size: 32px !important }`。
    *   **映射**: 建立 `1rem = 32px (设计稿)` 的心理映射。Tailwind 的 `p-4` (即 1rem) 对应设计稿的 `32px`。
3.  **视口安全 (Viewport Safety)**:
    *   **底部适配**: 底部固定元素（如底部支付坞）必须使用 `env(safe-area-inset-bottom)` 处理全面屏遮挡。
    *   **顶部对齐**: 自定义 Header 逻辑必须参考 `getMenuButtonBoundingClientRect()`，确保与微信胶囊按钮物理对齐。
4.  **禁手区与成熟实践 (Forbidden Zone & Best Practices)**:
    *   **单位规范 (Mature Practice)**:
        *   **SCSS 优先**: 所有静态尺寸（字号、宽高、间距）必须写在 SCSS 文件中。Taro 会在编译阶段将 `px` 自动换算为 `rpx`。
        *   **动态计算**: 若必须在 React `style` 中使用动态数值，**严禁直接写 px 字符串**。必须使用 `Taro.pxTransform(number)` 函数进行转换（例：`style={{ height: Taro.pxTransform(240) }}`）。
        *   **严禁硬编码 (Anti-Hardcoding)**: **除非万不得已，严禁在所有编码环节（逻辑、样式、配置）中出现硬编码行为。**
            *   **布局尺寸**：禁止在 WXS/JS 中硬编码高度、间距或偏移量（magic offsets）。必须优先通过 API (如 `getMenuButtonBoundingClientRect`) 或 `createSelectorQuery` 动态获取。
            *   **业务逻辑**：禁止使用魔法数字 (Magic Numbers) 或硬编码的 ID/枚举。所有此类值必须定义为常量或从配置接口获取。
            *   **极致鲁棒性**：编码必须允许“内容撑开”，确保在不同机型、不同文案长度下 UI 不崩坏。
        *   **物理像素**: 需要不随分辨率缩放的线条时，使用大写 `PX` 单位（例：`border: 1PX solid #000`）。
    *   **禁止手动 rem/rpx**: 禁止在 SCSS 或 JS 中手动写入 `rem` 或 `rpx` 后缀（Tailwind 自动生成的除外）。
    *   **禁止环境依赖**: 严禁引入 `window`, `document` 等浏览器对象。在处理微信与支付宝的差异时，**必须优先使用 Taro 提供的抽象接口**，保证双端兼容性。
    *   **资源与主题 (Assets & Themes)**:
        *   **SCSS 图片**: 严禁在 SCSS 的 `url()` 中引用本地资源。必须使用 Base64 (极小图标) 或远程 CDN URL。
        *   **CSS 变量**: 品牌色与状态色必须映射为 SCSS 变量或 CSS Variables，严禁在业务代码中直写 Hex 颜色值。

### 0.2 WXSS 兼容性与 Tailwind 安全规范 (WXSS Safety Rules)

由于微信小程序 (WXSS) 编译器不支持 CSS 选择器中的转义字符（反斜杠 `\`），必须严格限制 Tailwind 的使用方式：

1.  **禁手区 (禁止使用会触发转义的写法)**:
    *   **任意值 (Arbitrary Values)**: 禁止使用中括号，如 `h-[60px]`。
    *   **小数刻度 (Decimals)**: 禁止使用小数，如 `m-1.5`。
    *   **透明度斜杠 (Opacity Slashes)**: 禁止使用斜杠透明度，如 `bg-white/20`。
    *   **伪类/变体 (Variants)**: 禁止在类名中使用 `active:`, `hover:`, `group-active:` 等。
    *   **!important**: 禁止使用 `!p-4` 这种开头带感叹号的写法。
2.  **替代方案**:
    *   **动态变体**: 针对颜色、透明度、或通过 `Taro.pxTransform()` 计算后的动态偏移量，使用 React 行内样式 (`style={{ ... }}`)。
    *   **复杂 UI**: 涉及像素级还原的排版、动画、复杂伪类，统一在 `.scss` 文件中实现。
3.  **基础设施保护**:
    *   **全局选择器禁令**: WXSS 不支持 `*`。底层已通过 PostCSS 插件将其全量替换为 `page, view, text`。

### 0.3 测试规范 (Testing Specifications)

*   **测试策略**: 采用 **测试金字塔 (Testing Pyramid)** 策略。
    *   **Tier 1: 静态分析 (Static Analysis)** [100% 覆盖]
        *   **TypeScript**: 严格模式 (Strict Mode)，禁止隐式 Any。
        *   **ESLint/Prettier**: 零警告提交。
    *   **Tier 2: 单元与集成 (Unit & Integration)** [核心逻辑 80% 覆盖]
        *   **工具**: Jest + React Testing Library + Taro Mocks.
        *   **对象**: `utils/`, `hooks/`, 关键业务组件 (`Startup`, `Payment`).
        *   **快照测试 (Snapshot)**: 针对复杂且稳定的 UI 组件（如订单卡片）开启快照，防止意外布局崩坏。
    *   **Tier 3: 端到端与视觉 (E2E & Visual)** [关键路径覆盖]
        *   **工具**: 微信官方 `miniprogram-automator` + DevTools CLI 自动化通道（已落地）。
        *   **执行命令**:
            *   `npm run test:e2e`（仅执行端到端）
            *   `npm run test:e2e:weapp`（先构建小程序再执行端到端）
        *   **对象**: 核心赚钱流程 (扫码 -> 点单 -> 支付) 与首页核心资产交互（卡片堆叠/拖拽）。
        *   **视觉回归**: 针对设计还原度要求极高的组件（如首页卡片堆叠）进行像素级对比测试。

### 0.4 高级交互与动画规范 (Advanced Interaction)

1.  **渲染性能 (Animation Performance)**:
    *   **视图层驱动**: 涉及高频手势（如首页卡片推挤、拖拽、视差缩放）的交互，**必须**使用 WXS (WeChat) 或 SJS (Alipay) 实现。
    *   **禁手**: 严禁在 React 逻辑层实时更新受手势驱动的 Transform/Opacity 数值。
2.  **节点查询与 API**:
    *   **Taro API 优先**: 禁止使用任何平台私有 API (如 `wx.xxx`, `my.xxx`)。
    *   **DOM 替代**: 严禁使用 `document.querySelector`。必须使用 `Taro.createSelectorQuery()`。

## 第一章：页面层级结构 (Page Hierarchy)

本系统采用“单页应用 (SPA)” 架构，无传统 TabBar。所有核心入口折叠于首页。

*   **L0. 启动层 (私域独占启动)**
    *   **设计核心**：**去平台化 (De-Platforming)**。小程序启动后，必须让用户感觉这是 **"这家店的专属 App"**，严禁展示“附近门店列表”等公域分流入口。
    *   **冷启动逻辑 (Cold Start)**：
        1.  **若有历史访问记录**：直接加载 **[上次访问门店]** 的首页与数据。用户无感知这是一个平台。
        2.  **若由于外部扫码进入**：直接锁定并加载 **[码对应门店]**。覆盖历史记录。
        3.  **若为纯新用户 (无历史+无扫码)**：
            *   展示 **[空状态 - 扫码页]**。
            *   屏幕中央仅有一个 **[扫一扫]** 按钮与文案：“请扫描桌角或商家二维码，解锁专属美味”。
            *   *绝对不推荐“附近的好店”*。
    *   **LBS 仅作为辅助**：仅用于校验“是否在当前门店服务范围内”（防远程作弊），**绝不用于“找店”**。
*   **L1. 首页核心容器 (P00 首页)**
    *   **顶部导航**
        *   `Avatar` 头像 $\rightarrow$ **[P20 个人中心]**
    *   **店铺招牌**
        *   展示：商户 Logo、名称、Slogan、营业状态
    *   **卡片堆叠区**
        *   **[A] 口福红包卡 (口福红包)** (资产预览：Flow 堆叠) $\xrightarrow{Peek}$ **[P03 口福红包卡]**
            *   $\rightarrow$ **[P15 双模收银台 - 模式 B]** (卡片内点击具体票券亮码)
        *   **[B] 食福碎片卡 (食福碎片)** (资产预览：水平 Flow 堆叠) $\xrightarrow{Peek}$ **[P04 碎片卡]**
            *   $\rightarrow$ **[P10 碎银集市]** (管理中心内点击去获取)
        *   **[C] 寻味碎银卡 (寻味碎银)** (常驻显示：余额 & **今日步数**) $\xrightarrow{Peek}$ **[P01 碎银卡]**
            *   $\rightarrow$ **[P10 碎银集市]** (点击“逛集市”)
            *   $\rightarrow$ **[P11 游戏中心]** (点击“玩游戏”)
        *   **[D] 聚宝金库卡 (聚宝金库)** $\xrightarrow{Peek}$ **[P02 储值卡]**
            *   $\rightarrow$ **[P12 充值中心]** (点击 Banner)
            *   $\rightarrow$ **[P13 账单明细]** (点击“查看账单”)
    *   **策略活动区**
        *   **竖向滑动列表**：展示由商家确认发布的营销活动与资产通知。
    *   **底部支付坞**
        *   **设计原则**：**支付坞**。始终位于全局最高层级 (Fixed Bottom)，采用顶层悬浮设计。
        *   `支付按钮` $\rightarrow$ **[P15 双模收银台]**

## 第二章：首页与核心容器 (P00 首页)

### 2.1 整体布局
首页是所有功能的容器，采用**固定顶底 + 中间滚动**的结构。

| 区域 | 元素 | 描述 |
| :--- | :--- | :--- |
| **顶部固定区** | **顶部导航** | **左上头像**：显示用户圆形头像。**店铺简化名**：当店铺招牌不可见时，在头像右侧淡入显示店铺名（单行截断）。 |
| **可滚动区域** | **店铺招牌** | **定位**：紧贴顶部导航下方，参与页面滚动（非固定）。<br>**内容**：<br>1. 商户 Logo (圆形/方形)。<br>2. 商户名称 (大号加粗)。<br>3. 品牌 Slogan。<br>4. 营业状态标签。 |
| | **卡片堆叠区** | **定位**：位于店铺招牌下方。<br>**设计风格**：**统一高级浅色 (Premium Light)**。采用标准纸质感，辅以微妙的品牌色（靛蓝、石板、瑰红、蔚蓝）点缀。<br>**显示状态**：**折叠态**。所有卡片垂直堆叠，仅露出顶部 **60px** 标题区（额头）。 |
| | **策略活动区** | 位于卡片堆叠区下方。采用**竖向滑动列表**展示营销卡片与**任务进度引导**。 |
| **底部固定区** | **支付坞** | **水晶支付坞**：悬浮条。始终位于全局最高层级 (Fixed Bottom)，遮盖滑动列表内容。 |

### 2.3 核心动画：视差后退与物理遮挡 (Parallax Occlusion)

> **设计隐喻**：界面由三层不同高度的“资产层”组成。滚动时，顶层向下方的深处“塌陷”并由于移动速度差异被下层覆盖。

1.  **三阶段滚动流 (Three-Phase Flow)**：
    *   **Phase 1: 店铺招牌折叠 (Range: [0, Header_Height])**：
        *   **深度感知**：以底部为锚点 (`bottom-center`) 进行缩放（至 **0.70x**），营造向后方下沉的进深感。
    *   **Phase 2: 卡片堆叠收缩 (Range: [Header_Height, Stack_Fold_Threshold])**：
        *   **深度感知**：缩放至 **0.70x**，不透明度从 1.0 降低至 **0.5**（始终保持可见以维持遮挡感）。
    *   **Phase 3: 列表自由滚动 (Range: > Stack_Fold_Threshold)**：
        *   顶级资产已完全折叠/被遮挡，此时页面回归常规滚动模式。

2.  **内容吸引对齐**：
    *   区块间距（Gap）全局统一为 **40px**，确保视觉节奏稳健。
    *   随着卡片区域被遮挡，下方的**策略活动区**内容会平滑衔接，最终紧贴在**悬浮标题栏**下方。

3.  **恢复逻辑**：
    *   当用户回到页面顶部（`scrollTop` 趋于 0）时，所有层级按原路径视差弹回。

### 2.4 核心交互：卡片堆叠与点击聚焦 (Card Stack & Tap to Peek)

> **视觉隐喻**：界面如同手持的一叠银行卡。Card 3 (最顶层) 覆盖 Card 0 (最底层)。

1.  **堆叠态 (Default Stack)**：
    *   所有卡片垂直堆叠，默认仅露出 **60px** 额头高度。
    *   卡片层级关系固定，高索引卡片位于低索引卡片上方。


2.  **点击/聚焦 (Tap to Peek)**：
    *   **点击卡片** $\rightarrow$ **聚焦状态 (Peek)**：
        *   卡片进入聚焦态，其余卡片根据相对位置执行**垂直位移**（上方上推，下方下推），露出当前卡片的全部内容。
        *   **内容渲染 (Rendering)**：所有卡片内容**始终渲染于 DOM 中**，Tap to Peek 仅通过位移进行物理性露出/遮挡，不执行条件渲染。
        *   再次点击已聚焦的卡片或点击其他区域恢复默认堆叠态。
    *   **平滑过渡**：位移与恢复动作均使用平滑的动画过渡 (如 400ms cubic-bezier)。

3.  **细节打磨 (Micro-Interactions)**：
    *   **触感反馈**：点击选中卡片时提供轻微触感反馈。
    *   **安全区**：布局逻辑确保展开后的卡片底部不被水晶支付坞遮挡。

4.  **卡片比例**：银行卡标准 `85.60 : 53.98` (≈ 1.586)。

## 第三章：核心卡片 (Core Cards)

### P03. 口福红包卡
*   **激活与显示**：遵循 2.4 节 Tap to Peek 交互逻辑。
*   **功能**：快速预览、深度的资产管理、合成引导与社交互动。
*   **卡片定义**：**餐饮实物权证**。口福红包是碎片合成的最终成果，属于“准实物资产”。设计上突出厚度感与拟物感，是用户在餐厅进行价值兑付的唯一物理凭证。

#### 3.3 布局规范 (静态自适应布局)

> **核心约束**：严禁滑动或滚动。根据资产数量动态切换展示逻辑，始终保持卡片视觉完整性。

1.  **[A] 零资产态 (Empty State)**:
    *   **触发条件**：用户未拥有任何口福红包。
    *   **布局**：卡片正中央展示。
    *   **视觉**：半透明虚线圆形容器 + 拟物化“空餐盘”图标。
    *   **引导文案**：`“暂无可兑付资产”`。
    *   **行动点**：下方展示提示按钮 `[去合成第一道菜]`，点击触发 **Tap to Peek [P04 碎片卡]**。

2.  **[B] 孤品态 (Single Asset)**:
    *   **触发条件**：红包总数 = 1。
    *   **布局**：大图居中模式。
    *   **视觉**：展示该资产的高清 Isometric 素材，辅以全息动态光效（CSS 扫光）。
    *   **信息**：资产名称（加粗）、票面值（¥XX）。
    *   **交互**：点击资产区域 $\rightarrow$ 跳转核销。

3.  **[C] 简约态 (2-3 Assets)**:
    *   **触发条件**：红包总数 ∈ [2, 3]。
    *   **布局**：水平对称网格。
    *   **视觉**：资产图标缩小，并排展示。
    *   **特征**：显示简单的名称标签，突出“已就绪”感。

4.  **[D] 资产态 (4+ Assets)**:
    *   **触发条件**：红包总数 ≥ 4。
    *   **布局**：**2 x 3 固定网格**（最高展示 6 个槽位）。
    *   **表现逻辑**：
        *   前 5 个槽位展示最近获得的资产图标（带数量标 xN）。
        *   **第 6 个槽位 (聚合槽)**：展示一个虚化的“叠放图层”视觉，中心显示字样 `“+N”` 或 `“更多”`。
    *   **静态设计**：该卡片不提供展开列表。第 6 个槽位点击后直接跳转 **[P15 收银台-模式B]** 进行全量查看与核销，从而避开本卡片内的滑动需求。

### P04. 食福碎片卡
*   **激活与显示**：遵循 2.4 节 Tap to Peek 交互逻辑。
*   **功能**：资产深度预览、食谱切换、食材合成。
*   **卡片定义**：**过程性养成资产**。食福碎片是合成口福红包的中间原材料，通过养成式的收集（集市购买或步数转化）赋予用户游戏化的掌控感，是通往最终美食成果的必经之路。

#### 3.4 布局规范 (布局)
1.  **里程碑挑战 (产出源同步)**:
    *   **视觉表现**：在碎片卡内引入横向线性进度条，节点标注 `10,000` 与 `20,000` 步。
    *   **奖励掉落**：分别挂载 **[普通碎片]** (10k 📦) 与 **[稀有碎片]** (20k 💎)。
    *   **动态性说明**：奖励掉落受商户每日风控预算限制。若商户进入“经济模式”，奖励可能自动转换或门槛动态上浮。
    *   **交互反馈**：步数达标时节点高亮脉冲。点击已达标节点 $\rightarrow$ 触发碎片爆裂掉落动效，奖励存入资产预览区。
2.  **资产预览区 (水平 Flow 堆叠 - Peek 态可见)**:
    *   **表现形式**：展示当前拥有的所有碎片图标，采用**水平 Flow 流式堆叠**。

## 第四章：专项功能界面 (Specialized Functional Interaces)

### P10. 碎银集市 (集市)
*   **入口**：P01 碎银卡内点击 [逛集市]；P04 碎片卡内点击 [缺失食材]。
*   **设计原则**：**流动式纵向集市**。取消页签切换，采用单一滑屏容器，通过纵向滑动切换功能区。
*   **布局规范**：
    1.  **顶部：每日货架 (采购摊位)**:
        *   **每日特惠**：横向滚动展示 3 个随机普通碎片 (碎银计价)。
        *   **稀有黑市**：展示 1 个高亮稀有碎片 (高价/限量)。
    2.  **底部：我的背囊 (回购摊位)**:
        *   **表现形式**：位于采购区下方，下滑即可进入。
        *   **交互逻辑**：网格展示个人碎片库存。点击碎片弹出底部操作条，实时显示商行回收价，点击 **[出售]** 换取碎银。

### P11. 游戏中心 (游戏中心)
*   **设计原则**：**神秘街机厅 (Arcade Discovery)**。游戏不设预告，通过“盲盒”机制激发探索欲。
*   **入口**：P01 碎银卡内点击 [玩游戏]。
*   **交互规范**：
    1.  **探索发现 (盲盒态)**:
        *   界面展示一系列排列整齐的**虚拟盲盒**或**待机态街机位**。
        *   卡片呈现深色剪影，仅显示“？”或微弱的动效光环。
    2.  **解锁激活 (开盒感)**:
        *   **激活条件**：消耗碎银、达成特定食谱（入席令）、或随机触发掉落。
        *   **动效**：点击“开盒” $\rightarrow$ 播放拟真的“撕开包膜”或“屏幕亮起/系统初始化”动效，瞬间 Reveal 游戏封面与玩法快照。
    3.  **资产收藏**:
        *   已揭晓的游戏将色彩归位，正式作为“游乐资产”入驻。用户可通过**水平卡片滚动 (Horizontal Flow)** 快速启动。
    4.  **激励奖励与结算**:
        *   **基础奖励**：游戏分数按比例产出 `碎银` (如 1:100)。
        *   **卓越奖励 (高分掉落)**：随机掉落 **[食材碎片]**。掉落概率与门槛由商户侧风控 AI 根据实时预算动态动态微调。
        *   **产出闭环**：获得的碎片实时同步至 [P04 碎片卡]合成槽位中。

### P12. 充值中心 (充值中心)

### P13. 账单明细 (账单明细)
*   **入口**：P02 点击 [查看账单]。
*   **布局**：
    *   无限滚动列表。
    *   每行：左侧 (类型与时间)，右侧 (变动金额，绿进黑出)。
    *   类型涵盖：充值、消费、退款、游戏奖励、集市交易。
    *   **补开发票 (Invoice Request)**：
        *   针对“消费/充值”类条目，点击详情可看到 **[申请开票]** 按钮。

        *   系统根据 [Clawback 回溯算法] 自动计算可退金额（扣除已消费的赠送权益）。
        *   用户确认金额后提交，等待商户审核。

### P15. 双模收银台 (收银台)
*   **设计原则**：**拟物收银夹**。采用单页容器实现扫码与凭证的无缝切换，消除“切页感”。
*   **入口**：首页底部支付坞或外部商家码。
*   **核心交互**：
    *   **模式滑块 (Toggle)**：顶部或底部中心设置 **[扫码付款 | 会员凭证]** 轻量化胶囊滑块，点击或滑动进行原位状态切换。
    *   **模式 A (扫码付款 - 主动态)**：
        *   **交互逻辑**：
            1.  **输入总额**：用户输入消费总金额（如 ¥35.00）。
            2.  **智能抵扣 (Auto-Compute)**：
                *   系统瞬间计算**最优组合**：自动勾选 1 张 [招牌面入席令] (抵 ¥30) + 自动使用 [余额] (抵 ¥5)。
                *   UI 清晰展示：`总额 ¥35 - 券抵 ¥30 - 余额 ¥5 = 实付 ¥0`。
            3.  **确认支付**：一键完成核销与扣款。
    *   **模式 B (会员凭证 - 被动态)**：
        *   高亮展示二维码/条形码。
        *   **被扫逻辑**：商户扫码后，系统同样执行上述“智能抵扣”逻辑，直接在商户端完成扣款，用户端收到扣款通知。
